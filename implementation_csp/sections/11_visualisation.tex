\section{Visualisation et reporting interactif}

L'analyse manuelle des \texttt{data.json} étant fastidieuse, trois scripts produisent des
pages HTML statiques (autonomes, ouvrables en \texttt{file://} dans Chrome) qui permettent
d'explorer les résultats interactivement sans serveur web ni dépendance externe.

\subsection{\texttt{view.py} — viewer par taille}

Deux modes :
\begin{itemize}
  \item \textbf{Mode config} : \texttt{python view.py output/hX/config\_name} scanne le
    dossier et produit le \texttt{data.json} de cette config.
  \item \textbf{Mode agrégé} : \texttt{python view.py output/hX -{}-aggregate} consomme tous
    les \texttt{data.json} des sous-dossiers et produit \texttt{output/hX/view.html}
    embarquant les 8 configs (aucun \texttt{fetch}, aucune dépendance CDN).
\end{itemize}

Le viewer agrégé propose :
\begin{itemize}
  \item \textbf{Sélection de config} en haut (8 boutons), avec mode \textbf{comparaison}
    multi-sélection (panneaux côte-à-côte avec diff-highlight des molécules dont la
    classification diffère entre configs).
  \item \textbf{Cards de stats} : Molécules / Originaux plans / Solutions CSP / Solutions
    stables planes / Instables / Stables non planes (basculées selon présence du bloc
    \texttt{runs}).
  \item \textbf{Toolbar} : recherche par nom, filtres (Originaux, Solutions, Stabilité),
    expand all / collapse all.
  \item \textbf{Légende stabilité} repliable, expliquant les 6 classes et leurs critères.
  \item \textbf{Tableau des molécules} cliquables-dépliables. Pour chaque solution
    multi-runs : badge classification (couleur), \texttt{planar\_count/n}, $\mu \pm \sigma$,
    \textbf{barre de dispersion SVG} (échelle fixe 0--30°, ligne pointillée au seuil 10°,
    ticks pour chaque run individuel, marqueur $\mu$), valeurs \texttt{min--max}.
  \item \textbf{Section ``Couverture CSP''} repliable : analyse de la partition des solutions
    planaires entre les 8 configs (cf. ci-dessous).
  \item \textbf{Bandeau \texttt{batch\_meta}} : si \texttt{output/hX/batch\_meta.json}
    existe, affiche en haut les stats du dernier run \texttt{batch\_all} (durée, nombre
    d'instances, options).
\end{itemize}

\subsection{Section ``Couverture CSP''}

Permet de répondre à la question : \emph{quelles solutions chaque config trouve, lesquelles
sont communes, lesquelles sont uniques à un sous-ensemble} ? Deux visualisations côte-à-côte :

\begin{itemize}
  \item \textbf{Répartition par config (toutes solutions)} : un bar chart horizontal, une
    ligne par config, segmenté par classe de stabilité (mêmes couleurs que les badges).
    Permet de voir d'un coup d'\oe{}il quelle config trouve combien de solutions et leur
    profil qualité.
  \item \textbf{Liste d'intersections} (filtrée aux solutions \texttt{always\_planar} +
    \texttt{mostly\_planar}) : 1 section dépliable par sous-ensemble de configs, triée du
    plus universel (toutes les 8 configs) au plus unique (1 seule config). Cliquer dévoile
    la liste des couples \texttt{(molécule, sizes)} de cette intersection.
\end{itemize}

Identité d'une solution : couple \texttt{(mol\_name, sizes)}, indépendant de la config qui
l'a trouvée. Une solution est dite \emph{universelle} si elle apparaît dans les 8 configs,
\emph{unique} si elle n'apparaît que dans une seule.

\subsection{\texttt{update\_report.py} — rapport global}

Génère \texttt{csp\_solver/experiments/report/index.html}, un rapport unique couvrant toutes
les tailles \texttt{hX} testées. Contenu :

\begin{itemize}
  \item Navigation sticky avec \texttt{IntersectionObserver} pour le suivi de section active.
  \item Sections numérotées : Introduction, Méthodologie (pipeline + xTB + ACP), Résultats,
    Stabilité inter-runs, Observations, Détails, Commandes.
  \item \textbf{Charts SVG} générés en Python pur (pas de dépendance D3/Plotly) :
    \begin{itemize}
      \item Bar chart empilé : solutions planes vs non-planes par taille
      \item Donut chart : pourcentage global de solutions planes
      \item Pie chart : répartition des classes de stabilité
      \item Scatter plot $\mu \times \sigma$ : 1 point par solution, couleur par classe,
        tooltips avec $h$/config/molécule, ligne verticale au seuil 10°
    \end{itemize}
  \item \textbf{Tableau résultats par h} : nombre de molécules, solutions, planes, non
    planes, \% plan (avec barre de progression colorée), originaux plans.
  \item \textbf{Tableau stabilité par h} : 6 colonnes (1 par classe), affiché seulement si
    au moins une donnée multi-runs est disponible.
  \item Lignes dépliables : cliquer une ligne du tableau résultats déplie une mini-table par
    molécule.
\end{itemize}

\subsection{\texttt{bundle.py} — archive portable}

Crée \texttt{experiments\_bundle.zip}, archive autonome contenant :
\begin{itemize}
  \item \texttt{report/index.html} (rapport global)
  \item \texttt{output/hX/view.html} (viewer par taille)
  \item Les fichiers \texttt{*\_opt.xyz} référencés par les liens HTML (anciens single-run et
    nouveaux multi-runs, captés par 3 patrons de glob distincts)
  \item \texttt{README.txt} avec instructions d'ouverture
\end{itemize}

L'archive est destinée à être envoyée à un collaborateur. Le destinataire l'extrait et ouvre
\texttt{report/index.html} dans Chrome — toutes les visualisations fonctionnent en
\texttt{file://}, sans serveur ni installation.

\subsection{Affichage ``Aucune solution CSP''}

\dateblock{mai 2026 (mi-mois)}

Avec l'exclusion par défaut de la solution tout-hexagones et la régénération de la
table de voisinage par MD, certaines molécules (en particulier les benzénoïdes
entièrement gelés où chaque hexagone a $\deg=6$) se retrouvent avec \textbf{zéro

Trois éléments d'UI ajoutés dans \texttt{view.html} :

\begin{itemize}
  \item Sur la \textbf{ligne molécule} : la cellule \emph{Verdict} affiche une pill

    La cellule \emph{Solutions} montre \texttt{0} dans une étiquette grise. La
    ligne entière reçoit la classe CSS \texttt{.no-sol-row} qui grise le fond et
    italique-grise le nom — la molécule reste lisible mais clairement secondaire.
  \item Sur les \textbf{cards en haut} : si au moins une molécule est dans ce cas,
    une nouvelle card grise \texttt{$\oslash$ Sans solution CSP : N} apparaît
    automatiquement entre \emph{Solutions CSP} et \emph{Plans}. Cachée sinon
    (zéro pollution visuelle quand toutes les molécules ont des solutions).
  \item En \textbf{mode comparaison} (panneaux côte-à-côte) : même traitement,
    version compacte \texttt{$\oslash$ Aucune sol.} pour tenir dans des panneaux
    étroits.
\end{itemize}

Palette : \texttt{\#6a737d} / \texttt{\#eaecef} (gris neutre, distinct des verts /
oranges / rouges des verdicts pour ne pas créer d'ambiguïté).

\subsection{Section ``Couverture CSP'' alignée sur MD}

\dateblock{1\textsuperscript{er} mai 2026}

Le critère d'inclusion d'une solution dans la partition CSP était auparavant :
\textit{always\_planar + mostly\_planar} (multi-runs) avec fallback \texttt{sol.planar}.
Il devient simplement : \texttt{md\_validation.planar = true} (fallback \texttt{sol.planar}).

\textbf{Conséquence sur le bar chart ``Répartition par config''} : il n'affiche plus 6 classes
de stabilité mais 2 segments par ligne (vert plan / rouge non-plan), cohérent avec le
verdict MD. Les fonctions \texttt{computePerConfigBreakdown} et \texttt{renderPerConfigBarSVG}
ont été simplifiées en conséquence.

\subsection{Architecture des templates}

Les templates HTML/CSS/JS sont externalisés dans \texttt{templates/} pour bénéficier de la
coloration syntaxique de l'IDE et permettre l'édition propre. La substitution est faite côté
Python avec \texttt{string.Template} (syntaxe \texttt{\$variable}).

\begin{lstlisting}[style=benzai]
csp_solver/experiments/templates/
  common.css        # Palette + classes partagees (cards, planar/non-planar)
  view.html         # Template du viewer (substitution $h_name, $configs_json, ...)
  view.css          # Styles specifiques au viewer
  view.js           # Logique JS du viewer (IIFE avec etat prive,
                    # fonctions onclick exposees sur window pour le HTML inline)
  report.html       # Template du rapport global
  report.css        # Styles specifiques au rapport
  report.js         # Smooth scroll + IntersectionObserver
\end{lstlisting}

\textbf{Décisions architecturales} :
\begin{itemize}
  \item Le HTML est statique : toutes les données sont injectées au moment de la
    génération via \texttt{<script>window.\_\_DATA\_\_ = \{\ldots\};</script>}. Aucun fetch,
    aucune dépendance CDN, aucun build step.
  \item Le calcul interactif (filtres, tri, partition CSP, classification) est intégralement
    fait \textbf{côté JS} (à l'exception des charts du rapport global, générés en Python).
    Cela évite d'élargir le payload \texttt{data.json}.
  \item Les helpers SVG côté Python (\texttt{generate\_bar\_chart\_svg},
    \texttt{generate\_donut\_svg}, \texttt{generate\_pie\_svg},
    \texttt{generate\_scatter\_svg}) sont des fonctions pures retournant une chaîne
    \texttt{<svg>...</svg>}, faciles à tester et à réutiliser.
\end{itemize}

\subsection{\texttt{csp\_viewer} : refonte SQLite + Flask multi-datasets}

\dateblock{mai 2026 (fin de mois)}

\textbf{Motivation : le \texttt{view.html} monolithique ne passe pas h9.} Le viewer
décrit jusqu'ici charge l'intégralité de \texttt{data.json} en mémoire JS au démarrage.
Pour h3--h8 (de quelques dizaines à quelques milliers de solutions), c'est instantané.
Pour h9, la config \texttt{no-freeze\_no-table} contient à elle seule
${\sim}560\,000$ entrées (${\sim}600$~MB de JSON), et au total ${\sim}1{,}5$~M solutions
toutes configs confondues : le navigateur plante avant même de finir de parser le
fichier.

\textbf{Architecture retenue.} Une nouvelle implémentation sous
\texttt{csp\_solver/experiments/csp\_viewer/} adoptant le découplage
\textbf{stockage / API / présentation} :

\begin{itemize}
  \item \textbf{Backend SQLite} (\texttt{db\_all.db}) indexant l'arborescence
    \texttt{cluster\_results/h*}/ ou \texttt{csp\_solver/experiments/output/h*}/
    sans la modifier. Schéma dans \texttt{schema.sql}, PK = \texttt{(h, config, mol)}
    pour les molécules et \texttt{(h, config, mol, sol\_idx)} pour les solutions.
  \item \textbf{Serveur Flask local} (\texttt{server.py}, écoute sur
    \texttt{127.0.0.1:8765}) exposant une API JSON paginée avec filtres.
  \item \textbf{Frontend SPA} en JS vanilla : sélecteur de dataset, lazy-loading par
    requête API, pagination, filtres, loading screens. Aucune dépendance externe
    (sauf 3Dmol.js pour la 3D, cf.~\S\ref{sec:molviz}).
\end{itemize}

\textbf{Builder \texttt{build\_db.py}.} Multiprocessing : un worker par couple
(h, config, mol). Chaque worker lit le \texttt{md\_final\_opt.xyz} de chaque solution,
calcule la planarité ACP, repère les statuts \emph{geom\_infeasible} et
\emph{xtb\_failed} (cf.~ci-dessous), insère ses résultats dans la DB. Sur
ext4 + 30 c\oe{}urs côté cluster, un build complet h6--h9 prend ${\sim}5$~min ;
sur NTFS Windows, c'est plus lent à cause des dossiers à milliers d'entrées.

Mode \texttt{-{}-append} : permet de rebuilder \emph{une portion} de la DB
(ex.~juste h9) sans refaire les autres. Auto-detect via
\texttt{cluster\_results/\{h\}}, \texttt{csp\_solver/experiments/output/\{h\}}, ou
\texttt{\_\{h\}\_run/output/\{h\}} (pour exécution directe sur le cluster).

\textbf{Sémantique des compteurs.} Pour chaque couple (h, config, mol), quatre
catégories cohabitent :

\begin{itemize}
  \item \texttt{n\_solutions\_csp} : nombre de solutions trouvées par le solveur CSP
    (énumération combinatoire pure, sans contrainte 3D).
  \item \texttt{n\_md\_completed} : sols dont la reconstruction 3D \emph{et} la
    validation xTB ont réussi $\rightarrow$ pris en compte dans \texttt{n\_plans}
    et \texttt{n\_non\_plans}.
  \item \texttt{n\_geom\_infeasible} : sols \textbf{CSP-valides mais géométriquement
    inaccessibles}. La reconstruction lève \texttt{ValueError} (typiquement :
    pentagone/heptagone demandé sur un hexagone trop contraint, pattern
    \texttt{(1,1,1,1,1,0)}). Le \texttt{sol\_dir} est créé sur disque mais reste
    \emph{vide} (pas de \texttt{source.xyz}). C'est le comportement de
    \texttt{main.py} du run cluster --- pas un timeout, pas un bug, juste une
    impossibilité topologique.
  \item \texttt{n\_xtb\_failed} : sols dont la reconstruction a réussi
    (\texttt{source.xyz} présent) mais xTB n'a pas convergé. Très rare en
    pratique sur h6--h8, un peu plus fréquent sur h9 (structures très tendues).
\end{itemize}

L'invariant est :
\[
n\_solutions\_csp \approx n\_md\_completed + n\_geom\_infeasible + n\_xtb\_failed
\]

\textbf{Important pour h9} : avec les flags \texttt{no-freeze} et \texttt{no-table},
le CSP accepte une part importante de solutions géométriquement infaisables ---
jusqu'à ${\sim}75\%$ sur certaines mols. Ce \emph{n'est pas un manque de calcul} ;
ces sols ne sont juste pas physiquement réalisables. Le viewer les expose comme
lignes à part entière (badge jaune \emph{GÉOM~$\times$}, pas d'angle, pas de fichier
à servir) plutôt que de les masquer ou de les confondre avec des jobs incomplets.

\textbf{Endpoints API.} Tous filtrés par \texttt{h} :
\begin{itemize}
  \item \texttt{GET /api/datasets} : liste des h disponibles avec leurs compteurs.
  \item \texttt{GET /api/summary?h=hN} : stats par config.
  \item \texttt{GET /api/molecules?h=\&config=\&search=\&page=\&size=\&sort=}.
  \item \texttt{GET /api/solutions?h=\&config=\&mol=\&filter=\&page=\&size=\&sort=}
    avec \texttt{filter} $\in$ \{\emph{all, validated, plans, non\_plans, infeasible,
    xtb\_failed}\}.
  \item \texttt{GET /file?path=<chemin>} : sert un xyz/json/inp depuis le project
    root, avec \emph{path rewriting} automatique entre conventions cluster
    (\texttt{\_hN\_run/output/hN/...}) et locale
    (\texttt{cluster\_results/hN/...}), et \emph{glob fallback} sur
    \texttt{sol\_<idx>\_*} en cas de désync mineure du suffixe \texttt{sizes}.
  \item \texttt{GET /api/mol3d?path=<xyz>} : voir~\S\ref{sec:molviz}.
\end{itemize}

\textbf{Frontend SPA.} En haut du dashboard, un sélecteur de dataset
(\texttt{<select id="dataset-select">}) bascule entre h3, h4, \ldots, h9. Les cartes
configs montrent \emph{Solutions validées}, \emph{Géom. infaisables}, \emph{Plans},
\emph{Non plans}. La table des molécules a une colonne dédiée \emph{Géom.~$\times$}
qui chiffre les sols inaccessibles, et une colonne \emph{xTB échec} (visible
seulement si non-zéro). En cliquant une molécule, on voit la liste paginée de ses
solutions avec un dropdown filtre à 6 options.

\subsection{\texttt{molviz} : visualisation 3D des solutions}
\label{sec:molviz}

\dateblock{mai 2026 (fin de mois)}

\textbf{Objectif : voir la forme d'une solution sans quitter le navigateur.} Avant
ce module, ouvrir un \texttt{md\_final\_opt.xyz} demandait de télécharger le fichier
puis de l'ouvrir dans Avogadro --- friction réelle quand on parcourt des centaines
de solutions. Le module \texttt{molviz}, sous
\texttt{csp\_solver/experiments/csp\_viewer/molviz/}, ajoute un bouton \emph{3D} sur
chaque ligne de solution \texttt{plan} ou \texttt{non\_plan}, qui ouvre une modal
plein-écran avec un viewer 3D interactif.

\textbf{Bibliothèque retenue : 3Dmol.js.} ${\sim}200$~KB minified, drop-in via
balise \texttt{<script>}, support natif XYZ, API simple
(\texttt{addSphere}, \texttt{addCylinder}, \texttt{addLabel}). Mol* (molstar) et
NGL Viewer ont été écartés (overkill pour des molécules de 30--50 atomes), de
même que les approches 2D type SmilesDrawer/RDKit-JS qui demandent une conversion
XYZ$\rightarrow$SMILES non-triviale.

\textbf{Architecture du module.}

\begin{lstlisting}[style=benzai]
csp_solver/experiments/csp_viewer/molviz/
  __init__.py
  bonds.py        # detection liaisons C-C par distance
                  # + filtrage par cycles 5/6/7
  kekule.py       # matching maximum (Edmonds blossom)
                  # + identification des radicaux
  api.py          # blueprint Flask /api/mol3d + /molviz_static/
  static/
    molviz.js     # wrapper 3Dmol.js (rendu custom doubles, radicaux, cycles)
    molviz.css    # modal viewer
\end{lstlisting}

\textbf{Pipeline complet (côté serveur).} Sur appel à
\texttt{/api/mol3d?path=<xyz>} :

\begin{enumerate}
  \item Lecture du XYZ, on garde uniquement les carbones (les H ne sont pas affichés).
  \item Détection des liaisons par seuil de distance : C--C présente si
    $1{,}20 \leq d \leq 1{,}65$~\AA. Couvre toutes les longueurs réelles
    (double~$\sim$~1{,}34, aromatique~$\sim$~1{,}40, simple~$\sim$~1{,}54).
  \item Énumération exhaustive des cycles simples de longueur $\leq 7$ via
    \texttt{networkx.simple\_cycles(g, length\_bound=7)} (et non
    \texttt{cycle\_basis}, qui n'en retourne qu'une base dépendante du
    spanning tree).
  \item \textbf{Filtrage par taille} : on ne garde que les cycles de taille
    $\in \{5,6,7\}$ --- les seuls valides par construction de notre projet.
  \item \textbf{Filtrage des bonds} : on supprime ceux qui n'appartiennent
    \emph{à aucun} cycle retenu. Cible exactement les \emph{chord bonds}
    parasites qui apparaissent dans les structures fortement tendues
    (cf.~paragraphe suivant).
  \item Matching Kekulé (Edmonds blossom) sur le graphe nettoyé via
    \texttt{networkx.max\_weight\_matching(g, maxcardinality=True)}. Chaque arête
    couplée par le matching est étiquetée \emph{double bond}, les autres
    \emph{single bond}. Les sommets non-couverts par le matching sont les
    \textbf{sites radicalaires}.
  \item Sérialisation JSON : \texttt{atoms} (C uniquement),
    \texttt{bonds} (avec \texttt{order} 1 ou 2), \texttt{cycles}
    (avec \texttt{size} et liste d'atomes en ordre cyclique),
    \texttt{radicals} (indices d'atomes), \texttt{meta}
    (compteurs et flag \texttt{perfect\_matching}).
\end{enumerate}

Le tout sous cache LRU de 256 entrées : recalculer un Kekulé prend ${\sim}10$~ms
par molécule, le cache sert les requêtes répétées en $<1$~ms.

\textbf{Le piège des chord bonds parasites.} Sur des structures très tendues
(ex.~4 pentagones + 4 heptagones d'h9), la déformation finit par rapprocher deux
carbones non-adjacents dans le squelette CSP à une distance $< 1{,}65$~\AA.
La détection par seuil voit alors une fausse liaison, qui crée à son tour un
faux 3-cycle. Sans correction, le rendu affichait un triangle parasite au c\oe{}ur
de la molécule. Les étapes 4--5 ci-dessus l'éliminent : tout 3-cycle est rejeté
par le filtre $\{5,6,7\}$, et le bond parasite (qui n'apparaît que dans le
3-cycle) est supprimé en cascade. Effet de bord positif : le matching Kekulé
travaille désormais sur le \emph{vrai} squelette du non-benzénoïde, donc les
radicaux qu'il identifie ne sont plus pollués par un sommet parasite.

\textbf{Rendu côté client (\texttt{molviz.js}).} En tant qu'IIFE exposant
\texttt{window.MolViz.open(solInfo)}. Hooké par \texttt{app.js} sur le bouton
\emph{3D} de chaque ligne de solution.

\begin{itemize}
  \item \textbf{Atomes} : sphères grises de rayon 0{,}22~\AA. Radicaux en violet
    (\#9333ea) avec rayon 0{,}32 et label \og{}$\bullet$\fg{} flottant au-dessus.
  \item \textbf{Bonds simples} : un cylindre de rayon 0{,}07~\AA{} entre les deux
    centres atomiques.
  \item \textbf{Bonds doubles} : deux cylindres parallèles décalés perpendiculairement
    de 0{,}10~\AA{} dans le plan de la molécule.
  \item \textbf{Cycles colorés} : polygone semi-transparent (opacité 0{,}45) coloré
    par taille --- pentagone orange clair (\#ffe4b5), hexagone gris (\#e5e7eb),
    heptagone bleu clair (\#bfdbfe). Triangulé manuellement via centroïde +
    \texttt{addCustom} car 3Dmol.js n'a pas de polygone natif.
\end{itemize}

Trois toggles dans la barre du modal : \emph{Cycles colorés}, \emph{Radicaux},
\emph{Indices atomes}. Légende des couleurs en bas à droite. Header avec compteurs
(\texttt{N C}, \texttt{K liaisons (D doubles)}, perfect matching ou
\texttt{N radicaux}). Fermeture par \texttt{Esc}, clic sur l'overlay ou bouton
\texttt{$\times$}.

\textbf{Choix : matching topologique vs.~longueurs xTB.} Les doubles bonds sont
posés par \emph{matching topologique} (sémantique chimique propre, identifie
naturellement les radicaux), pas par longueur de liaison effective. C'est cohérent
avec les analyses futures Kekulé/Clar (qui partent du graphe, pas des distances).
Une variante par longueur peut être ajoutée plus tard (drapeau API
\texttt{?bonds=distance}) pour comparaison visuelle.

\textbf{Évolutions prévues}.
\begin{itemize}
  \item Tables annexes en DB pour propriétés topologiques :
    \texttt{kekule\_structures}, \texttt{clar\_indices}, \texttt{nics\_values},
    liées par \texttt{(h, config, mol, sol\_idx)}. Calculées par scripts dédiés
    (topologiques rapides ou via xTB/DFT selon la grandeur), insérées via
    \texttt{INSERT OR REPLACE} pour rester idempotentes.
  \item Vue de comparaison cross-dataset (h $\times$ config $\times$ propriété)
    sur la même page.
  \item Toggle \emph{double bonds par longueur xTB} dans le modal pour comparer
    matching topologique vs.~géométrie réelle.
\end{itemize}

\subsection{Navigation parmi les structures de Kekulé}
\label{sec:molviz-kekule-nav}

\dateblock{mai 2026 (mi-mai, post-réunion chimistes)}

\textbf{Motivation.} Après échange avec Denis Hagebaum-Reignier et Yannick
Carissan, l'intérêt chimique central des structures 5/7 est l'étude de la
\emph{délocalisation des électrons} : où peuvent-ils se placer, comment se
répartissent les sites radicalaires, et comment varient ces configurations
selon les Kekulé possibles. La vue par défaut du \texttt{molviz} ne montrait
qu'\emph{un} matching maximum arbitraire ; on a besoin de pouvoir parcourir
\emph{toutes} les Kekulé d'une molécule pour visualiser ces alternatives.

\textbf{Architecture : un système de modes dans le modal.} Le modal a maintenant
une barre de modes (chips \emph{Défaut} / \emph{Kekulé}) au-dessus des toggles
existants. La vue \emph{Défaut} conserve le comportement initial (matching max).
La vue \emph{Kekulé} fait apparaître une barre de navigation
\og{}$\blacktriangleleft$\quad $i / N$\quad $\blacktriangleright$\fg{} et
re-rend la molécule avec les liaisons doubles / radicaux du $i$-ème Kekulé
de la liste énumérée. Touches clavier $\leftarrow$ / $\rightarrow$ pour
naviguer. Architecture extensible : les futurs modes \emph{Clar} et
\emph{RBO} viendront s'ajouter comme nouvelles chips sans refactor.

\textbf{Convention de vocabulaire : Kekulé vs Radicalaire.} Au sens strict
chimique, une structure de Kekulé requiert un couplage parfait (chaque carbone
porte exactement une liaison double). Pour les molécules à nombre impair de
carbones ou à topologie qui force des électrons non appariés (\emph{concealed
non-Kekuléan}, typique des structures à cycles 5/7), aucun couplage parfait
n'existe : on parle alors de \emph{configurations radicalaires} plutôt que de
structures de Kekulé. L'algorithme énumère dans les deux cas les matchings de
cardinalité maximum --- seul le nom affiché change. La chip de mode s'appelle
\emph{Kekulé} quand \texttt{n\_radicals == 0}, \emph{Radicalaires} sinon, avec
une icône d'aide \og{}ⓘ\fg{} qui explique la situation au survol. C'est une
question de rigueur de vocabulaire, pas d'algorithme.

\textbf{Backend.} Énumération canonique via backtracking
(\texttt{kekule.enumerate\_kekule}) sur le squelette carbone :

\begin{enumerate}
  \item Calcul de la cardinalité $M$ d'un matching maximum
    (\texttt{nx.max\_weight\_matching}). Budget de radicaux $= n - 2M$.
  \item Backtracking déterministe : à chaque étape on prend le sommet
    d'index minimal non encore décidé, et on branche sur (a) le matcher
    avec chaque voisin libre (ordre croissant des voisins), (b) le laisser
    comme radical si le budget le permet.
  \item Plafond \texttt{max\_count} (défaut 200, max 1000) avec retour
    \texttt{(liste, is\_exact)} où \texttt{is\_exact=False} signale qu'on
    a coupé avant épuisement.
\end{enumerate}

L'ordre canonique garantit que chaque Kekulé est généré une et une seule
fois (pas de déduplication nécessaire). Comportement byte-déterministe :
même molécule $\to$ même liste, dans le même ordre.

\textbf{API.} Nouvel endpoint \texttt{GET /api/kekule\_list?path=<xyz>\&max=N}
qui renvoie \texttt{\{kekule: [\{bond\_orders, radicals, n\_doubles\}, ...],
meta: \{returned, is\_exact, has\_more, max\_requested\}\}}. Cache LRU 256
entrées avec clé \texttt{(xyz, max\_count)}, lazy-loadé par le frontend
uniquement lors du premier clic sur \emph{Kekulé}.

\textbf{Tests.} Le script
\texttt{molviz/test\_kekule\_enum.py} vérifie le compte de Kekulé sur des
molécules de référence (benzène = 2, naphtalène = 3, anthracène = 4,
phénanthrène = 5), l'absence de doublons dans l'énumération, et l'invariant
\og toutes les Kekulé ont la même cardinalité \fg{}. Test additionnel sur
le mécanisme de \texttt{max\_count} et le flag \texttt{is\_exact}.

\textbf{Plan des étapes suivantes.} L'architecture est posée pour ajouter
sans refactor :
\begin{itemize}
  \item un mode \emph{Clar} : nouvelle chip, nouveau module \texttt{molviz/clar.py},
    nouvel endpoint \texttt{/api/clar} ;
  \item un mode \emph{RBO} : idem avec \texttt{molviz/rbo.py} et
    \texttt{/api/rbo}, rendu sous forme de heatmap par cycle.
\end{itemize>
